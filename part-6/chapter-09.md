# Chapter 9: Handoff、模式提取、FDE 的 T 字成长

## 9.1 交接不是丢代码，是交判断

fde-book 第六部分（精进）讲一个反直觉的点：**最好的 FDE 交接，是让客户团队不再需要你。** 交接的成败不看你留下多少代码，看客户能不能自己判断"系统好不好、坏了怎么修、新需求怎么加"。

所以交接物有三层，越往下越值钱：

```
第一层（最浅）：代码 + 部署文档
   ——客户能跑，但不知道好坏

第二层（中间）：评估集 + 跑分脚本
   ——客户能判断"系统是不是工作"（铁律二的交接形态）

第三层（最深）：整套 harness + 判断逻辑
   ——客户能自己加功能、自己守铁律（方法论 Harness 交出去了）
```

## 9.2 三层交接物清单

把前面所有章节的产出汇成交接清单：

| 交接物 | 来自章节 | 让客户能做什么 |
|---|---|---|
| `evals/*.jsonl` + 跑分脚本 | Ch3-4 | 自己判断系统好坏、监控回归 |
| `.github/copilot-instructions.md`（三条铁律 + 安全护栏） | Ch2,5,8 | 团队一致地守规矩，加功能不破坏铁律 |
| `.github/agents/` + `.github/prompts/` | Ch5-6 | 用同一套专家视角和工作流开发 |
| `skills/`（客户 SOP/手册） | Ch7 | agent 按客户规程做事，可自行扩充 |
| Spec-Kit 配置 | Ch6 | 规格驱动地复现交付新功能 |
| Azure 部署 + 可观测 + 成本/回滚 | Ch8 | 自己运维、灰度、出问题归因 |
| `docs/discovery.md` + SOW 过线分 | Ch3 | 记得当初为什么这么做（判断的出处） |

**关键**：这套东西全在 git 里，全可审计，全带"为什么"。交接 = 把仓库的所有权移交，加一次"怎么用这套 harness"的培训。

## 9.3 模式提取：一个项目的判断，喂回你的方法论

fde-book 精进章的另一面：每个项目结束，FDE 要做**模式提取**——把这次踩的坑、做对的判断，沉淀成下次能复用的东西。

对应到工程，就是把项目里反复出现的判断固化成可复用资产：

- 这次客户的"工单分诊 SOP"做成了 skill → 下个类似客户能改改就用
- 这次发现"等保合规要把数据流约束写进 Rule 层" → 进你的 Rule 层模板库
- 这次的安全护栏段落 → 进你的客户启动包（附录 C）

**这正是本书的来历**：fde-book 是别人提取的方法论模式，workshop 是你提取的工程模式，这本书是把两份模式焊起来的元提取。你做的每个 Copilot 落地项目，都该这样往回沉淀。

## 9.4 FDE 的 T 字成长

fde-book 收尾讲 FDE 的成长形状是 **T 字**：

```
─────────────────────  横：广度
   懂业务、懂客户、懂多个技术栈、懂怎么跟人对齐 Outcome
        │
        │  竖：深度
        │  至少一个领域扎得很深（你的是 Copilot/Azure 生态的 harness 工程）
        │
```

- **竖**：你在微软生态的 harness 工程上要扎到底——三层 harness、Spec-Kit、Azure 部署、安全沙箱，比谁都熟。
- **横**：但你不能只懂 Copilot。你要懂客户的业务（保险、制造、零售）、懂怎么把技术翻译成 Outcome、懂 AWS/Azure/GCP 的对应物（这样客户问"和 Bedrock 比怎样"你答得上来）。

**给微软合作伙伴技术支持的具体建议**：
- 竖的方向：把这本书的每个工件在真实客户项目里跑一遍，遇坑就更新 Rule 层模板和启动包。
- 横的方向：保持对 fde-book 这类跨平台材料的阅读——它的 AWS 判断框架能帮你在客户做平台对比时不露怯。

## 9.5 收尾：回到那个星期一

回到前言里合作伙伴群的那个问题——"Copilot 像高级补全、Agent 一进合规就卡死，到底怎么落地"。

走完这本书，你的答案不再是一句话，是一套时间线：

1. **第一周**：别急着配 agent，先观察工作流、和客户定一个 Outcome 数字、建 30 条评估集、SOW 写进过线分。（Ch3）
2. **第二周起**：搭三层 harness，把三条铁律固化成 `.github/` 工件，渐进上 agent。（Ch2,5）
3. **PoC**：手作 handoff 让客户看见每个专家判断，靠评估分过线，不靠 demo 运气。（Ch6,8）
4. **生产化**：升 Spec-Kit、过安全沙箱、上 Azure（私网/Key Vault/可观测/回滚）、每个 demo 标成本。（Ch6,8）
5. **交接**：把评估集 + 整套 harness + 判断逻辑全交出去，让客户不再需要你。（Ch9）

`Outcome = Harness × Customer`。你能调的是 Harness，两层都搭好，Outcome 自然来。
