# 附录 C: 第一周客户启动包

新接一个 Copilot/Azure 落地项目，第一周照这个清单走。这是 fde-book "第一周模板" + workshop 企业最佳实践的焊接版。

## C.1 第一周日程（建议）

| 天 | 重点 | 产出 |
|---|---|---|
| D1 | 权限 + 观察 | 拿到仓库/环境权限，开始看工作流 |
| D2 | 观察 + 对齐 Outcome | 定 1 个 Outcome 数字 |
| D3 | 评估集冷启动 | 真实 10 条 + 刁难 10 条 |
| D4 | 客户挑刺 + 补样本 | 评估集 v0 共 30 条 |
| D5 | Rule 层 + SOW | `copilot-instructions.md` 草稿 + SOW 过线分 |

## C.2 第一周必争取的权限（拖到第三周会被当"不该你有"）

- [ ] 客户仓库：至少能开 PR，最好 main 的 PR 合并权（受 CI 保护）
- [ ] Azure 订阅：staging 环境的部署权
- [ ] 一个 hot fix 上线通道（Fix Forward 的前提）
- [ ] 客户业务方一个固定联系人（评估集挑刺、人工打分要靠他）

## C.3 对齐 Outcome 的提问模板

❌ 别问："你们想要什么功能？"
✅ 要问：
- "现在这个事最慢/最痛的环节是哪个？"（找 Outcome 锚点）
- "如果只能改善一个数字，你最想看到哪个变好？"（收窄成单一 Outcome）
- "这个数字现在是多少？你觉得做到多少算成功？"（定基线和过线分）

客户答不出具体数字 → 还在 Discovery，别开始搭 harness。

## C.4 评估集 v0 冷启动（4 步）

1. 客户已有数据抽 **10 条真实样本**（给不出 → 还在 Discovery）
2. 自己写 **10 条刁难样本**（口语化/缩写/错别字/多步推理/"我不知道"类）— 可让 Copilot 起草，你审定
3. 客户业务方**看分布挑刺**（关键！让他认账）
4. 按挑刺方向**再加 10 条**

存进 `evals/<domain>_v0.jsonl`，第一周内 commit。

## C.5 Rule 层第一版（最小）

第一周不配 agent，但 Rule 层先立起来：
- 技术栈约定（跟客户现有代码一致）
- 三条铁律段（过线分填 SOW 谈定的数字）
- 安全护栏段（合规团队会问，先备好）

骨架见附录 B.2。

## C.6 SOW 必含项

- [ ] 1 个明确的 Outcome 指标 + 当前基线 + 目标值（= 过线分）
- [ ] 过线分 = `evals/` 里的 min_score（合同数字和仓库数字一致）
- [ ] 验收方式：评估集机器分 + 业务专家人工抽检
- [ ] 交接范围：代码 + 评估集 + harness + 一次培训（附录见 Ch9）

## C.7 第一周红灯（出现就停下重新对齐）

- 🚩 客户说不出一个具体 Outcome 数字 → 还在 Discovery
- 🚩 客户连 10 条真实样本都给不出 → 数据没准备好
- 🚩 客户要"AI 驱动的全公司知识库" → 目标太宽，收窄
- 🚩 第一周拿不到任何环境权限 → Fix Forward 没法做，先谈权限
